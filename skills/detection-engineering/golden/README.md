# Golden fixtures — provenance and limits

These 58 files are the repo's snapshot test fixtures. They are imported by the
ingest, detect, view, and pack tests under each layer. This README states
explicitly what they are and — more importantly — what they are not, so nobody
mistakes a passing snapshot for a passing precision/recall claim.

## Provenance

**Every file in this directory is synthetic.** No file was captured from a
production cloud, identity provider, MCP runtime, or warehouse. Each fixture
was hand-crafted to:

- match the upstream vendor's documented schema as closely as the public
  documentation describes,
- exercise one positive and (where applicable) one negative path through the
  skill that consumes it,
- be small enough to read top-to-bottom in a code review.

When the consuming skill is migrated onto a new wire-format version
(OCSF 1.7 → 1.8, MITRE ATT&CK v13 → v14, etc.), the corresponding
`*_sample.ocsf.jsonl` is regenerated by running the ingester against the
`*_raw_sample.*` input and snapshot-checking the result back into this
directory. That is why the OCSF outputs are versioned alongside the raw
inputs — the snapshot proves a *contract*, not a *capture*.

## What these fixtures verify

- The ingester accepts the documented vendor schema.
- The detector / converter is deterministic for that input.
- The OCSF envelope is valid against the OCSF JSON Schema.
- Re-running the same input produces byte-identical output.

## What these fixtures do NOT verify

These are **not** evidence of any of the following claims:

- "X% precision / Y% recall on real workloads."
- "We've seen this attack pattern in the wild."
- "Production-scale event volumes pass through cleanly."
- "Our detector beats the false-positive rate of vendor Z."

Whenever a downstream document (README, blog post, scorecard) wants to make
a claim of that shape, it must cite a *different* source — a captured-fixture
corpus, a customer telemetry sample, or a public dataset — and not this
directory.

## Roadmap toward real fixtures

The sibling [`../captured/`](../captured/README.md) directory now exists and
holds the first batch of public, license-clean attack-traffic samples. It
has the **opposite** provenance contract from this directory:

- `golden/`   — synthetic, hand-crafted; verifies wire contract + determinism.
- `captured/` — real public traces under permissive licences; verifies that
  detectors fire on actual attack-pattern traffic.

Still planned but not yet shipped:

- Filling out `captured/` for the remaining detectors. Tracked in
  [issue #420](https://github.com/msaad00/cloud-ai-security-skills/issues/420);
  the directory's `README.md` Roadmap section keeps the live gap list.
- Per-detector precision/recall scoring on the captured corpus, posted as a
  PR comment. Tracked in
  [issue #419](https://github.com/msaad00/cloud-ai-security-skills/issues/419).

Until those land, every claim about real-world detector quality should
explicitly say "synthetic fixtures only" — and so should you. If a claim
needs to cite captured traffic, point at `captured/` and not this
directory.

## Adding a fixture

1. Add the `*_raw_*` input under this directory.
2. Run the consuming ingester / detector locally and pipe its stdout into the
   matching `*_sample.*` snapshot file in this directory.
3. Update the consuming skill's test to load the new fixture.
4. Open a PR. The CI snapshot tests will lock the file's content from then on.

## Citing this directory honestly

If you reference these fixtures in a doc or PR description, prefer:

> "Tested against a synthetic golden fixture in
> `skills/detection-engineering/golden/`. Does not yet include a
> captured-traffic corpus (#420)."

Avoid:

> "Tested in production." (untrue — no production data has touched this
> directory.)
> "Tested at scale." (untrue — fixtures are deliberately small.)
> "Validated against real attacker traffic." (untrue.)
